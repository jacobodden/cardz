Storage Persistence Refactor — Comparison

Evaluating three architectures for ⛐-local: Current (Express + SQLite) vs localStorage vs IndexedDB

Prepared for planning — July 2026

📚 Current — Express + SQLite

Server-side SQL via better-sqlite3

✅ Pros

  • Data Authoritative server-side data with ACID transactions
  • Data Foreign keys, CHECK constraints, data integrity
  • Data SQL queries with indexes — efficient, flexible
  • Infra Multi-device & multiplayer ready (one DB, many clients)
  • Infra Concurrent access handled by the server
  • Infra Backup/restore: just copy cardz.db
  • Perf Large datasets (100k+ rows) are fine
  • Dev Already built, tested, and working

❌ Cons

  • UX Requires a running server — no offline use
  • UX Every action = HTTP round-trip + network latency
  • Perf State is re-fetched on every navigation (no cache)
  • Infra Deployment is more complex (server process, port, CORS)
  • Dev Need to maintain both client and server code
  • Arch Game logic split across server (scoring) and client (UI)

⚠️ Risks if unchanged

  • Zero offline capability — app is dead without the server
  • Growing API surface as features are added
  • Session management, auth, and concurrency complexity if scaled

🗃️ Option A — localStorage

Client-side key-value (strings)

✅ Pros

  • Dev Dead simple API: getItem / setItem / removeItem
  • Dev Synchronous — no async/await, no callbacks
  • Dev No dependencies, no build step, no server changes
  • UX Works fully offline
  • Perf Near-zero latency for reads/writes (in-process)
  • Infra Eliminates the entire Express server for persistence
  • Dev Trivial to inspect in DevTools

❌ Cons

  • Data 5–10 MB limit — risky for sessions with many rounds/scores
  • Data Strings only → JSON.stringify every read/write (2–5× overhead)
  • Data No transactions — partial writes corrupt state silently
  • Data No indexes or queries — must load & scan everything in JS
  • Perf Synchronous — blocks the main thread on every access
  • Arch Single-device only — data is tied to one browser
  • Arch Browser can clear data without warning (storage pressure, user action)
  • Dev Manual migration logic — no versioned schema
  • Infra Eliminating the server means losing the single source of truth

⚠️ Key risk

  • Storage limit will be hit with moderate use (e.g. 10 sessions × 8 players × 20 rounds = ~1600 score rows)
  • No way to recover data once localStorage is cleared
  • Major rewrite of all data-access code in the client

🗃️ Option B — IndexedDB

Client-side NoSQL (structured)

✅ Pros

  • Data Large storage (usually ≥50 MB, often unlimited)
  • Data Stores structured data directly — no JSON serialization overhead
  • Data Indexes for efficient querying (e.g. all rounds for session X)
  • Data ACID transactions across multiple object stores
  • Perf Asynchronous — does not block the main thread
  • UX Works fully offline
  • Dev Built-in versioned schema (migrations via onupgradeneeded)
  • Arch Object stores map naturally to existing DB tables
  • Infra Can eliminate the server, or keep it as a sync endpoint

❌ Cons

  • Dev Complex API — raw IndexedDB is callback-heavy and error-prone
  • Dev Realistically needs a wrapper (Dexie.js, idb) → added dependency
  • Dev Steeper learning curve than localStorage
  • Arch Single-device only — same limitation as localStorage
  • Arch No built-in server sync — must build if multi-device needed later
  • Perf Transaction overhead for small, simple writes
  • UX Quiet quota failures — QuotaExceededError requires handling
  • Dev Harder to debug than localStorage (though DevTools has improved)
  • Infra If server is removed, there’s no cloud backup of user data

⚠️ Key risk

  • API complexity makes the migration itself risky — thorough tests are essential
  • Still single-device unless a sync layer is added later

📚 Current (SQLite)

★★★☆☆

Best for multi-device, data integrity, and scaling. Worst for offline, simplicity, and deployment overhead.

🗃️ localStorage

★★☆☆☆

Best for trivial, small-scope data (theme, prefs). Not recommended for game sessions, rounds, and scores — storage limits and lack of transactions are dealbreakers.

🗃️ IndexedDB

★★★★☆

Best for offline-first client-heavy apps. Viable replacement if multiplayer is not a priority. Recommended over localStorage for this domain.

🔧 Migration Impact Assessment

localStorage

  • Files touched: ~10+ (all pages, api/client.ts, possibly game logic)
  • Server changes: Could remove server entirely (or keep for sync later)
  • New deps: None
  • Testing: Rework all integration tests; add storage-mock tests
  • Risk: High — storage limit and no transactions are fundamental limitations

IndexedDB

  • Files touched: ~10+ (same scope as localStorage, minus server elimination)
  • Server changes: Could eliminate the server or keep as optional sync layer
  • New deps: 1 (recommended: idb or dexie)
  • Testing: Rework integration tests; use fake-indexeddb for unit tests
  • Risk: Medium — API complexity, but fundamentally capable

📜 Recommended approach

Option B (IndexedDB) is the better choice if the goal is a truly offline-first, client-heavy app. It avoids localStorage’s hard limits and transaction problems, and maps well to the existing relational model. Use the idb library (npm i idb) to get a Promise-based API that hides IndexedDB’s callback complexity.

However, the current Express + SQLite architecture is not broken. If multiplayer or multi-device sync is ever a goal, keeping the server and adding IndexedDB as a local cache layer (for offline support) is a stronger long-term strategy than a full migration to either client-only option.

Recommendation: Add IndexedDB as an offline cache in front of the existing API, rather than replacing the server wholesale. This gives offline capability without sacrificing the server as a source of truth.
