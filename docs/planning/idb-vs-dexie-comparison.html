idb — minimal promise wrapper

by Jake Archibald · ~1.5 KB gzipped
4/5 — Simplicity & size

✅ Pros

  • Size Tiny — adds negligible bundle weight (~1.5 KB gzipped)
  • API Mirrors native IndexedDB — no magic, zero surprises
  • DX Just adds await to the native API — nothing to learn
  • DX TypeScript-first — excellent type inference
  • FT Full access to all IDB features (indexes, cursors, ranges, transactions)
  • FT Works with any build tool, no config needed
  • Risk Low risk — thin wrapper, unlikely to break or need updates

❌ Cons

  • API Verbose — still need to open stores, manage transactions manually
  • API No query builder — must use IDB ranges and indexes directly
  • DX No schema DSL — schema is defined in onupgradeneeded
  • DX No live queries / reactivity — manual re-fetch on writes
  • DX More boilerplate for complex queries (compound sorts, filters)

dexie — full-featured DB wrapper

by David Fahlander · ~16 KB gzipped
5/5 — Fit for app-level data

✅ Pros

  • API Rich query builder — .where(…).sortBy(…), compound queries
  • API Declarative schema with db.version().stores() — clean migrations
  • DX liveQuery() — Observable-powered reactivity for React
  • DX Built-in transaction helpers — db.transaction('rw', …)
  • DX Table filtering & collection API — .filter(), .and(), .toArray()
  • DX Schema versioning & migration callbacks built in
  • FT db.cloud if sync is ever needed later (optional addon)
  • DX Excellent documentation and community examples

❌ Cons

  • Size ~10× larger than idb (~16 KB vs ~1.5 KB gzipped)
  • API More abstraction — magic can hide what IndexedDB is actually doing
  • Risk Slightly higher maintenance risk — bigger API surface, more bug surface
  • Risk Lock-in — switching from Dexie later means rewriting the data layer
  • DX Live query unsubscribes must be managed to avoid memory leaks

📈 Recommendation: dexie for cardz-local

Why Dexie wins for this codebase:

• The data model has relational cross-references (sessions → players, rounds → scores). Dexie’s query builder and compound indexes make these lookups trivial; with idb you’d be manually managing IDB index ranges and cursor iteration.

• Schema versioning is essential — as the app evolves, table schemas will change. Dexie’s db.version(N).stores({...}) + .upgrade() is far cleaner than manually wrangling onupgradeneeded version-switch blocks.

• React integration. Dexie’s useLiveQuery hook directly replaces the current pattern of useEffect + fetch + setState. Every page currently re-fetches on mount — Dexie makes that automatic and incremental.

• Bundle size is not a constraint in this project — ~16 KB gzipped is negligible for an app that already loads React, React Router, etc. (well over 100 KB gzipped).

• Will not need server sync — the db.cloud optional addon is irrelevant here, but it’s nice that it exists if the decision ever changes.

When would you pick idb instead?

If you needed to bundle under ~50 KB total, or if the data layer was very simple (single object store, no relational queries), or if you wanted maximum control and minimal abstraction. For a game session manager with relational data and React integration, Dexie is the better fit.
